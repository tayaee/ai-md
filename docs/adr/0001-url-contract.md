# ADR-0001: URL 계약 — PHP식 1:1 매핑 + 파일 단위 서브 마운트

상태: 승인 (2026-07-12)

## 맥락

초안 명세에 내부 모순이 있었다. 라우팅 룰은 "`.ai.md` 확장자가 보이는 URL"을
정의했는데(`/src/convert.ai.md`), 시나리오 B 본문은 확장자 없는
"`POST /convert` 엔드포인트 개설"을 요구했다.

## 결정

`.php`와 같은 해석 모델을 채택한다: **`.ai.md` 파일 하나 = 미니 앱 하나**,
파일 경로가 URL prefix가 되고, 명세 본문의 라우트는 그 prefix **하위에** 마운트된다.

- `GET /tetris.ai.md` → 컴파일된 테트리스 SPA HTML
- `GET /convert.ai.md` → 해당 API 앱의 Swagger UI(`/docs`로 302)
- `POST /convert.ai.md/convert` → 명세 본문이 정의한 실제 엔드포인트
- `GET /` → `public/index.html` 정적 서빙 (아래 개정 참조)
- 그 외 (`.ai.md`인데 `src/`에 대응 파일 없음) → `404`

(2026-07-14 1차 갱신) 최초에는 `index.ai.md` 자체가 테트리스 SPA였으나, 랜딩
페이지 요구가 생기면서 테트리스는 `tetris.ai.md`로 옮기고 `index.ai.md`는 두
데모로 가는 랜딩 페이지 전용으로 재정의했었다 (`GET /` → `/index.ai.md` 302).

(2026-07-14 2차 갱신 — 이 계약을 대체) "`.ai.md` 파이프라인은 URL이 `.ai.md`
확장자를 가질 때만 적용된다"는 원칙을 더 분명히 하기 위해, 루트(`/`)를 아예
이 파이프라인 밖으로 뺐다. `/`는 이제 nginx가 `public/index.html`을 곧바로
정적 서빙하며, 리다이렉트도 없고 엔진(main.py)도 이 요청을 보지 않는다
(엔진에 직접 `GET /`를 보내도 `_AIMD_RE` 불일치로 404). `index.ai.md`라는
이름의 명세/아티팩트는 더 이상 존재하지 않는다 — 랜딩 페이지는 LLM이 컴파일한
결과물이 아니라 손으로 작성한 순수 정적 자산이다.

## 결과

- 파일 1:1 매핑 철학과 명세 본문 라우팅이 모순 없이 공존한다.
- URL에 `.ai.md`가 그대로 노출되어 POC의 데모 가치(자연어 파일이 곧 앱)가 산다.
- FastAPI 서브앱의 자동 `/docs`가 prefix 하위에서 공짜로 나온다.
- `.ai.md` 확장자가 없는 경로(루트 포함)는 전적으로 nginx의 일반 정적 서빙
  규칙을 따르고, 엔진/LLM 파이프라인과는 무관하다.
